Intent Validation Design (AVA-Gen)¶

This document explains how intent validation works in the AVA-Gen runtime: how we decide whether a user’s request is supported for a given app, how we map it to a VA method, and how that drives ActionPlan selection.

1. Goals¶

Intent validation in AVA-Gen should:

  • Constrain user requests to what each app actually supports.
  • Reuse the skills descriptions you already generate (per-method).
  • Support multi-turn conversations while still blocking topic shifts.
  • Be backend-agnostic (OpenAI now, other models later).
  • Return structured results so the runtime can clearly branch:
  • unsupported → clarification
  • supported but ambiguous → clarification
  • supported + concrete method → action plan

2. Data Sources: skills → intents + intent→method map¶

For each app, we start from a skills description file under workspace/:

workspace/
  skills_description/
    {app_id}_skills_description.json

Typical format (context_methods variant):

{
  "app_id": "hu.vmiklos.plees_tracker",
  "context_methods": {
    "accessStatistics": {
      "description-short": "Open sleep statistics view",
      "description-detail": "Opens the app's statistics screen to view summarized sleep data and averages."
    }
  }
}

From this, IntentInterpreter derives two runtime artifacts under workspace/intent/:

  1. Intent list – for GPT only
[
  {
    "app_id": "hu.vmiklos.plees_tracker",
    "intents": [
      "Open sleep statistics view",
      "Opens the app's statistics screen to view summarized sleep data and averages."
    ],
    "intent_summary": "Summarizes the main capabilities of the app in one short sentence."
  }
]

Stored at: workspace/intent/intent_list_full.json

  1. Intent → method map – for server only
{
  "hu.vmiklos.plees_tracker": {
    "Open sleep statistics view": "accessStatistics",
    "Opens the app's statistics screen to view summarized sleep data and averages.": "accessStatistics"
  }
}

Stored at: workspace/intent/intent_method_map.json

Key separation:

  • GPT sees only intent_list_full.json (strings).
  • The runtime uses intent_method_map.json to resolve matched_intent → method_name.

3. Core Component: IntentValidator¶

File: interpreter/intent_validator.py

3.1 Responsibilities¶

  • Load all *_skills_description.json files from workspace/.
  • For each app_id, build:
  • intents: List[str] – allowed intent descriptions.
  • intent_to_method: Dict[str, str] – mapping intent → method_name (when available).
  • Provide a runtime API to:
  • Expose allowed intents for each app.
  • Validate a user message against those intents using a pluggable backend.

3.2 Public API¶

class IntentValidator:
    def get_intents_for_app(self, app_id: str) -> List[str]: ...
    def get_method_for_intent(self, app_id: str, intent: str) -> Optional[str]: ...
    def validate(self, app_id: str, message: str, history: Optional[List[Any]]) -> IntentValidationResult: ...

IntentValidationResult¶

@dataclass
class IntentValidationResult:
    is_supported: bool
    matched_intent: Optional[str] = None
    method_name: Optional[str] = None  # always resolved server-side from intent→method map
    reason: Optional[str] = None

4. Backend Interface: IntentMatcherBackend¶

To keep the system flexible, the actual “AI” piece is abstracted as a backend:

class IntentMatcherBackend(Protocol):
    def match_intent(
        self,
        app_id: str,
        message: str,
        intents: List[str],
        history: Optional[List[Any]] = None,
    ) -> IntentValidationResult:
        ...

Key design choices:

  • The backend must only choose among the given intents, or reject.
  • It cannot invent new intents or switch apps.
  • It may look at a small history snippet (last few turns) to resolve pronouns, but cannot expand scope beyond the provided intents.

4.1 OpenAI Backend (intent-only schema)¶

OpenAIIntentMatcherBackend implements IntentMatcherBackend using the project-local openai_client.send_request_to_gpt.

Schema used for GPT responses:

class IntentMatchResultModel(BaseModel):
    is_supported: bool
    matched_intent: Optional[str] = None
    reason: Optional[str] = None

Key points:

  • GPT never returns a method_name.
  • matched_intent must be exactly one of the provided intent strings (or null).
  • reason is a short explanation (at most ~20 words) used for clarifications.

send_request_to_gpt requests this schema and parses the model output as JSON (with some robustness against fenced ```json blocks).

5. Conversation Flow Integration¶

File: runtime/agents/conversation_agent.py

ConversationAgent is the “brain” that uses IntentValidator and ActionPlanStore.

5.1 High-level flow¶

On each /agent/request:

  1. Load Session from SessionStore.
  2. Append a user turn:
Turn(role="user", message=message, type="user", timestamp=...)
  1. If intent_validator configured:
response = self._handle_with_intent_validator(session, message)

Otherwise:

response = self._fallback_clarification(session, message)
  1. Save the updated session.
  2. Return AgentResponse to the caller (clarification or action_plan).

6. Intent Validation Logic in ConversationAgent¶

6.1 _handle_with_intent_validator(...)¶

Steps:

  1. Call the validator:
result = intent_validator.validate(
    app_id=session.app_id,
    message=message,
    history=list(session.turns),
)
  1. Extract:
is_supported  = result.is_supported
matched_intent = result.matched_intent
reason         = result.reason

  1. Case 1 – Not supported (is_supported == False)

  2. Build a clarification message explaining that the request is not supported for this app (using reason if available).

  3. Append a server turn with type="clarification_unsupported".
  4. Keep session.status = OPEN.
  5. Log event: intent_not_supported.

  6. Return:

    AgentResponse(type="clarification", ...)

  7. Case 2 – Supported: resolve method via map

  8. Always resolve method_name from the server-side map, never from GPT:

    method_name: Optional[str] = None
if result.matched_intent:
    method_name = intent_validator.get_method_for_intent(
        app_id=app_id,
        intent=result.matched_intent,
    )

  9. If we cannot resolve method_name:

    • Build a configuration clarification:

      “I recognized your intent, but I could not map it to a concrete action. Please double-check the intent→method configuration.”

    • Append server turn with type="clarification_ambiguous".
    • Log event: intent_ambiguous.
    • Return AgentResponse(type="clarification", ...).

  10. Case 3 – Have method_name: load ActionPlan

  11. Attempt:

    plan = actionplan_store.get_actionplan(app_id, method_name)

  12. If plan is missing:

    • Build a no-plan clarification message.
    • Append type="clarification_no_plan".
    • Log event: actionplan_missing.
    • Return AgentResponse(type="clarification", ...).

  13. Case 4 – Success: ActionPlan available

  14. We now have a valid ActionPlan and method.

  15. Construct a user-facing summary:

    • If matched_intent is available, use it as the short message:
    server_message = matched_intent
    • Otherwise fall back to:
    server_message = f"Executing method '{method_name}' for app '{app_id}'."

  16. Append server turn with type="action_plan".

  17. Set session.status = ACTION_SENT.
  18. Log event: actionplan_selected.

  19. Return:

    AgentResponse(
  type="action_plan",
  message=server_message,
  method_name=method_name,
  action_plan=plan,
  next_session_id=None,
)

7. Preventing Topic Shifts¶

Key rules that prevent the user from drifting away from the app’s domain:

  1. Fixed app_id per session

  2. Every session is tied to exactly one app_id.

  3. All validation uses that app’s intents.

  4. Closed intent set

  5. IntentValidator.validate always passes a fixed list of intents for this app.

  6. The backend must only choose among these or say “not supported”.

  7. Per-turn validation

  8. Every new user message in the session is validated again.

  9. If the user suddenly asks “book me a flight”, that won’t match any plees_tracker intents → is_supported=False → clarification.

  10. History is limited

  11. Only a small snippet of recent turns is provided to the backend.

  12. History is used for disambiguation, not for expanding to new capabilities or apps.

8. Session Lifecycle & Intent¶

  • SessionStatus.OPEN
  • No action plan has been sent yet.
  • User can go through multiple rounds of clarification.
  • SessionStatus.ACTION_SENT
  • An ActionPlan has been chosen and returned to the client.
  • The current session is “consumed” for that VA request.
  • Client can start a new session for the next task.

9. Summary¶

  • skills_description → intent strings + intent→method mapping.
  • IntentValidator → central interface for:
  • listing allowed intents per app
  • validating messages with a backend
  • IntentMatcherBackend → pluggable model layer (OpenAI or others).
  • ConversationAgent → uses validation to:
  • enforce app scope
  • handle unsupported / ambiguous cases with clarifications
  • produce ActionPlans when intent is solid.

This gives you:

  • Clear control over what each app supports.
  • A clean separation between:
  • data (skills + actionplans),
  • inference (intent backend),
  • runtime logic (ConversationAgent).
